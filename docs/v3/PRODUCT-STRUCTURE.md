# 深圳高薪岗位职场资讯库 V3：产品结构

## 1. 产品不再是什么

V3 不是一份把岗位按 30K、50K、100K 排列起来的高薪技术报告，也不是一套替用户直接选工作的推荐系统。

薪资档位只用于汇总观察。用户进入岗位证据库后，始终看到 Boss 直聘原始薪资区间、公司、岗位名称和对应岗位详情页。

## 2. 产品真正帮助用户完成的事

把招聘平台从“投简历的地方”变成“研究市场正在为什么付钱的资料库”。用户沿着四个问题阅读：

1. **最近出现了哪些值得留意的新岗位？**
2. **企业为这些岗位购买的能力组合是什么？**
3. **如果只选一个岗位，企业的共同要求是什么？**
4. **哪些方向同时具有需求、薪资和扩招信号？**

这四个问题依次对应：发现新职业、拆解需求、精准对标、赚钱地图。岗位证据库是四个页面共同引用的底层证据，不与它们并列争夺注意力。

## 3. 全站信息架构

### 3.1 首页：先说明怎么研究，而不是先堆图表

首页只承担三个任务：

- 用一句话建立方法：**别急着投简历，先看看市场正在高价购买什么。**
- 用四步阅读路线解释产品能做什么。
- 用 4—5 个真实研究切片引导用户进入深页，而不是一次展示全部结论。

首屏不再用静态生成图片。后续视觉实现采用已经确认的 Axion Studio 技术方向：实时 Shader、流体橙色光带、玻璃折射、胶囊导航、文字滚动按钮和移动端底部菜单。Shader 只服务于首屏氛围，不承载数据含义。

首页模块顺序：

1. Hero：核心问题、数据日期、主入口“开始逆向搜索”。
2. 方法说明：四步法的非术语化解释。
3. 研究切片：新岗位、能力重组、单岗对标、扩招信号、机会位置各一张。
4. 证据声明：样本规模、核验比例、不能得出的结论。

### 3.2 发现新职业

回答“市场里出现了哪些过去不常见、但现在已有多家公司付高薪招聘的岗位”。

每个岗位条目必须包含：

- 归一化岗位名与 Boss 原始岗位名变体；
- 样本数、公司数、行业数；
- Boss 原始薪资分布；
- 为什么判定为新兴信号；
- 旧能力如何重新组合；
- 所有证据岗位直链。

正式榜单目标为 30 个岗位。少于 3 条岗位或少于 2 家公司的方向，只能放在“早期信号”，不能伪装为成熟岗位。

### 3.3 拆解需求

回答“企业写在岗位要求里的高频能力与能力组合是什么”。

用户先选择一个明确岗位，再看到：

- 完整岗位要求样本数和样本公司数；
- 能力出现频率与原文证据；
- 两项、三项能力的共同出现关系；
- 经验、学历、语言、行业背景门槛；
- 不同真实薪资区间的能力差异；
- “新职业 = 旧能力重新组合”的具体案例。

只有存在可核验完整岗位要求的样本才参加频率统计。列表页摘要只能用于发现岗位，不能混入正式能力占比。

### 3.4 精准对标

回答“如果只选这一个岗位，我应该按企业需求补什么”。

首个正式对标岗位不由主观指定，而是从数据中选择同时满足以下条件的岗位：

- 至少 50 条可核验的同岗岗位要求；
- 来自多家公司，而不是单一企业批量发布；
- 有清楚的薪资梯度；
- 能力要求能够稳定归类。

页面展示必备能力、加分能力、高薪附加能力、常见工作成果、代表公司与原文证据，并倒推出学习优先级。这里输出的是“企业要什么，先学什么”的路径，不扩展成 30 天打卡计划。

### 3.5 赚钱地图

回答“哪些岗位方向同时具备需求、薪资和可进入性”。

- 横轴：市场需求，以去重后的岗位数、公司数和扩招信号共同表示；
- 纵轴：Boss 原始薪资上限的稳健统计值；
- 气泡大小：有效岗位样本数；
- 颜色：能力门槛；
- 点击气泡：展开真实岗位、能力缺口与相邻成长方向。

扩招模块展示同一企业重复招聘的岗位、数量、日期和原始链接。真正的薪资趋势必须建立在不同日期的重复快照上；只有一个日期时明确显示“正在积累下一次快照”，不使用上涨、下降等措辞。

### 3.6 岗位证据库

岗位库是所有结论的可追溯底座。筛选器保持克制，只保留：行业、区域、真实薪资、排序。搜索作为辅助入口。

每条记录展示：Boss 原始岗位名、公司、原始薪资、区域、行业、经验、学历、采集日期和证据状态；“查看来源”必须直接打开这家公司当前岗位的 Boss 详情页。同名岗位由多家公司发布时全部保留。

### 3.7 方法与来源

集中解释采集边界、去重、岗位归一化、能力标签、薪资统计档位、扩招信号、时间快照、证据等级和局限。专业术语使用小字号注释，不抢占主要阅读路线。

## 4. 全站导航

桌面端：`发现新职业 / 拆解需求 / 精准对标 / 赚钱地图 / 岗位证据库`

移动端保留同样的五个入口，使用底部抽屉呈现，不删减功能。

## 5. 内容密度规则

- 每个页面只回答一个主要问题。
- 首屏最多一个核心结论、一个解释段和两个动作。
- 图表默认只展示一个结论，其余信息通过点击或展开查看。
- 页面正文不连续堆放超过两个复杂图表。
- 统计口径和证据等级放在相邻的小字号注释中。
- 数据不足时使用明确的“尚不能判断”，不使用空图表填满页面。

## 6. V3 验收问题

任何一个页面上线前，都要能用一句话回答：

- 用户为什么进入这页？
- 看完后能做出什么选择？
- 结论能否回到具体岗位原文？
- 手机上是否不需要横向拖动就能完成主要任务？
