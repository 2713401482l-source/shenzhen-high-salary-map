# V3 数据契约与分析门槛

## 1. 三层数据

### 发现层：岗位线索

用于发现岗位名称、公司、薪资和市场覆盖。列表页记录可以进入这一层，但必须标明尚未核验详情。

### 证据层：岗位详情

用于能力统计、单岗对标和岗位库。必须能够证明公司、岗位、深圳地点、Boss 原始薪资和岗位要求来自同一岗位详情页。

### 时间层：重复快照

同一岗位在不同日期的观察分别保存，不覆盖旧记录。时间层用于判断岗位持续存在、企业重复招聘和薪资变化。

## 2. 岗位观察记录

每次采集形成一个 observation，核心字段如下：

| 字段 | 含义 |
| --- | --- |
| `observationId` | 本次观察的唯一编号 |
| `snapshotId` | 采集批次，格式 `YYYY-MM-DD-序号` |
| `bossJobId` | Boss 岗位编号 |
| `sourceUrl` | 当前公司岗位详情页 |
| `capturedAt` | 实际采集时间 |
| `evidenceLevel` | 列表观察、列表+详情、完整详情 |
| `title` / `company` | Boss 原始岗位名与公司名 |
| `salaryText` | Boss 原始薪资文本，不改写 |
| `salaryMin` / `salaryMax` | 月薪区间数值，单位 K |
| `salaryMonths` | 明示的薪数，未明示则为空 |
| `district` / `addressText` | 深圳区域与地址证据 |
| `industry` | 公司行业 |
| `experience` / `education` | 原始门槛文本 |
| `descriptionText` | 完整岗位职责与要求 |
| `headcountSignal` | 招聘人数或批量招聘原文信号 |
| `publishedAt` | 平台明示发布日期；无则为空 |

禁止保存账号、Cookie、聊天、联系方式或其他私人信息。

## 3. 归一化记录

原始字段永久保留，分析层新增：

- `normalizedRole`：精确岗位方向，例如“海外 TikTok 运营”，不能只写“运营”。
- `roleFamily`：较宽的岗位族群，仅用于导航和汇总。
- `emergingStatus`：`established`、`emerging`、`early-signal` 或 `insufficient`。
- `skills`：标准能力标签。
- `skillEvidence`：每个标签对应的岗位原文、岗位 ID 和来源链接。
- `responsibilityOutcomes`：企业希望该岗位交付的工作结果。
- `analysisEligibility`：允许参加哪些分析，以及被排除的理由。

## 4. 薪资口径

- 岗位证据库永远展示 `salaryText`。
- 30K / 50K / 100K 只作为汇总档位，按月薪区间中位数互斥归档：`<40K`、`40K–<75K`、`≥75K`。
- 仅当岗位上限触达约 30K 时进入研究范围，因此 `25K–35K` 可以进入 30K 档。
- 年薪、日薪、时薪和无法换算的薪资不进入月薪样本。
- 多薪、奖金、期权单独显示，不擅自折算进月薪。

## 5. 结论门槛

| 结论 | 最低门槛 |
| --- | --- |
| 新兴岗位正式条目 | 3 条岗位、2 家公司，并有至少 1 条详情证据 |
| 早期岗位信号 | 1—2 条真实岗位，明确标注小样本 |
| 能力频率 | 只统计完整详情证据，分母公开 |
| 能力共现 | 同一岗位要求中共同出现，至少 3 次后展示 |
| 精准对标 | 同一归一化岗位 50+ 条完整岗位要求，且公司来源分散 |
| 企业扩招 | 同一快照多岗位或跨快照重复招聘，展示岗位与日期证据 |
| 薪资变化 | 至少两个不同日期的可比快照；一个日期不称为趋势 |

## 6. 采集目标

- 发现池：至少 300 条去重后的深圳高薪岗位线索；
- 新兴岗位：30 个归一化方向，成熟信号与早期信号分开；
- 详情证据：优先把高价值方向的岗位要求补全；
- 精准对标：至少 1 个岗位方向达到 50+ 条同岗详情；
- 时间快照：每次采集追加新批次，为后续薪资和扩招变化保留历史。

如果平台验证阻止连续采集，不绕过验证，不编造样本；保留已经核验的数据，转入清洗、归类、分析和产品实现。

## 7. 可复跑分析产物

正式页面不在浏览器里临时拼结论。执行 `npm run v3:analyze` 后生成 `data/v3/analysis/real-analysis.json`，它是 V3 真实视图的统一输入。

产物包含：

- 每个岗位方向的样本数、公司数、行业数、薪资分布、详情证据和岗位直链；
- 仅基于详情核验样本的能力频率、能力共现及证据岗位；
- 单岗对标距离 50 条详情要求还差多少；
- 同一快照里的企业多岗位信号，明确标记为截面聚集而不是持续扩招；
- 不同采集日期数量，少于两个日期时禁止生成薪资涨跌结论；
- 发现池、详情证据、新兴方向、单岗对标和时间快照的正式版准备度。

`npm run v3:analysis-check` 会反查每个证据岗位是否存在于真实数据层，确认能力分母没有混入列表摘要，并检查所有发布门槛。
